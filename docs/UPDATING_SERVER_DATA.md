# Updating server data

Three ways to get server data onto the LXC, from most-preferred to
last-resort:

1. **Pull from git** — versioned, one-click updates. See §Git below.
2. **Dropzone / archive upload** — drag files or a folder or a `.tgz` into
   the Files tab; the archive is unpacked server-side. Best for one-off
   uploads and world imports.
3. **SFTP direct** — bulk transfers where the browser is too slow (multi-GB
   world imports). Least automated but always works.

## What "server data" means here

| Kind | Where it lives | Best way to update |
|---|---|---|
| Server binary (`server.jar`, `PalServer`, Forge tree) | `install_dir` | **Minecraft**: upload jar via Files tab (or Git). **SteamCMD**: Control → **Update Game Software**. **Forge**: upload tree via Files tab (or Git). |
| Mods, plugins, resource packs | `install_dir/mods`, `plugins`, `Pal/Mods` | Drop the folder into the Files tab, or a `.tgz` of the whole mods dir. |
| Config (`server.properties`, `PalWorldSettings.ini`, `user_jvm_args.txt`) | `install_dir` | Files tab: download, edit locally, upload with overwrite. Or use the Git flow. |
| World / save files | `world_dir` (on TrueNAS) | Files tab area=`world`, OR restore a backup tarball, OR SFTP for multi-GB imports. |
| Backups | `worlds/_backups/<name>/` | Auto-generated by Backups tab. |

## Small (< ~500 MB): dropzone

1. Server detail → **Files** tab.
2. Pick **area**: `install` (binary/config/mods) or `world` (saves).
3. Set **Save under** if you want everything to land in a subdirectory
   (e.g. `mods`).
4. Drop files onto the dropzone or use the three pickers:
   - **+ Files** → multi-select.
   - **+ Folder** → whole directory tree, `webkitRelativePath` preserved
     (drop `mods/` → files land as `mods/<name>.jar`).
   - **+ Archive** → single `.zip` / `.tar.gz` / `.tgz` / `.tar` / `.tar.bz2`,
     unpacked server-side.
5. Toggle **overwrite existing** if you're replacing.
6. Toggle **extract archives on the server** off if you'd rather upload the
   archive as a single file instead of unpacking it.

Per-file **progress rows** below the dropzone go green on success, red with
the error on failure. Uploads are sequential.

Same three pickers appear in the **New Server** modal under **Initial files
(optional)** — anything you queue there uploads right after the server def
is created, so you can pre-populate at creation time.

## Big trees: SFTP

Multi-GB world imports or thousands of mod jars — browser is slow. SFTP
into the LXC as the `gamesrv` user (or as your admin user + `chown` after):

```bash
sftp gamesrv@10.0.0.204
cd /opt/gamesrv/worlds/minecraft-smp/
put -r /local/path/to/world
```

Then in the manager, if you replaced the whole world, **Restart** the
server. Ownership check:

```bash
sudo -u gamesrv test -w /opt/gamesrv/worlds/minecraft-smp && echo "gamesrv can write ✓"
```

(Per the Bot Manager NFS note: test *behaviour*, not displayed owner — NFS
UID mapping can lie.)

## Git: versioned + repeatable

Best for anything you'd already keep in source control (server pack, config
files, custom launcher). See [ADDING_SERVERS.md](ADDING_SERVERS.md) for the
`git_source:` schema. Quick tour:

1. Server detail → **Git** tab.
2. Fill **Repo URL**, **Ref** (branch/tag/commit; blank = default), and
   any of **Subdir in repo** / **World subdir** / **Extra excludes**.
3. For private repos: paste a fine-grained PAT (Contents: **Read only**)
   into **PAT for this sync** — not persisted anywhere.
   Alternative: set the PAT in `/etc/gamesrv.env` under any name (e.g.
   `MC_FORGE_GIT_PAT=github_pat_xxx`) and set **Token env var name** to
   `MC_FORGE_GIT_PAT`.
4. **Save Git Source** → **Pull & deploy**.

The `.git` cache lives at `<install_dir>.gitsrc/` — the runtime install dir
never sees `.git`. The rsync into `install_dir` uses `--delete-excluded`,
so files matching an exclude pattern are pruned but hand-uploaded files not
in the repo are preserved.

**Update available** on the Git tab: **Check remote** does `git ls-remote`
(no clone) and shows a chip if the remote has moved past your deployed
SHA. **Pull & deploy** again to update.

## Replacing the Minecraft server.jar

1. **Files** → area `install`, drop the new `server.jar` with **overwrite**
   ticked (or leave off if you want a `.part` collision error to remind you
   what you're doing).
2. **Control** → **Restart**.

Optional: back up the world first (**Backups → Back up world_dir now**) —
some updates migrate the world format irreversibly.

## Patching Palworld (or any SteamCMD game)

1. **Control** → **Update Game Software** — re-runs
   `steamcmd +app_update 2394010 validate`, which patches in place.
2. **Restart** if the server was running.
3. Saves live in `world_dir` (symlinked to `Pal/Saved`), so they survive
   updates untouched.

## Editing a config file

1. **Files** → area `install`, path e.g.
   `Pal/Saved/Config/LinuxServer/PalWorldSettings.ini`.
2. Click **download**.
3. Edit locally.
4. Upload again with the same path and **overwrite** ticked.
5. **Restart** the server for changes to load.

## Importing a world from another host

Two options:

- **Tarball route (recommended for anything under ~2 GB):** on the old host,
  `tar -czf world.tgz world/`. In the New Server modal or the Files tab
  → **+ Archive** with area `world` → the archive streams to the LXC and
  gets unpacked in one round trip. Safe: staging dir first, tar members
  checked for escape/symlink.
- **SFTP direct** (multi-GB): SFTP the world dir straight into `world_dir/`
  as above.

## Safety rails baked in

- Every file-API call is scoped to `install_dir` or `world_dir` and rejects
  paths that escape via `..` or absolute paths (see `app/uploads.py`
  `_safe_join`).
- Uploads stream to `<name>.part` and are atomically renamed.
- Archive extraction stages into a scratch dir first, checks every member
  for path-escape / symlink / device / hardlink, then merges into the
  target. `overwrite=true` on the extract endpoint requires a
  `dest_subdir` — you can't nuke the area root by accident.
- **Restore** requires the server to be stopped (409 otherwise) and
  refuses tarball entries that would escape the staging dir. Existing
  world is renamed to `.<name>.replaced-<ts>` rather than deleted, so an
  accidental restore is undo-able.
- Deleting the server root is refused explicitly.
- Backups live under `worlds/_backups/<name>/` — on TrueNAS, so ZFS
  snapshots protect them for free.
