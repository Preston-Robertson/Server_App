# Updating server data

Preston asked specifically about being able to update server data through the
app itself. This is the full playbook.

## What "server data" means here

| Kind | Where it lives | How to update |
|---|---|---|
| The game binary (`server.jar`, `PalServer`, …) | `install_dir` | **Minecraft**: upload new jar via UI. **SteamCMD**: **Update Game Software** button. |
| Mods, plugins, resource packs | `install_dir/mods`, `install_dir/plugins`, `install_dir/Pal/Mods` | Upload via UI. |
| Server config (`server.properties`, `PalWorldSettings.ini`) | `install_dir` | Download → edit → upload with overwrite. |
| World / save files | `world_dir` (on TrueNAS) | Upload individual files via UI OR restore a backup tarball OR SFTP for large trees. |
| Backups | `worlds/_backups/<name>/` | Auto-generated by **Backup** button. |

## Small files (< ~500 MB): use the UI

1. Server detail → **Files** tab.
2. Pick **area**: `install` (binary/config/mods) or `world` (saves).
3. Set **path**: relative path inside the area, e.g. `server.jar`,
   `mods/mymod.jar`, `Pal/Saved/Config/LinuxServer/PalWorldSettings.ini`.
4. Choose the local file, tick **overwrite** if replacing.
5. Upload. The file streams to `<path>.part` then atomically renames — a
   dropped connection leaves no corrupt file behind.

Download works the same way (list → click **download** on the file's row).

## Big files or whole trees: use SFTP

For multi-GB world imports or bulk mod folders the browser is slow. SFTP into
the LXC as the `gamesrv` user (or as your admin user + `chown` after):

```bash
sftp gamesrv@10.0.0.203
cd /opt/gamesrv/worlds/minecraft-smp/
put -r /local/path/to/world
```

Then in the manager, if you replaced the whole world, **Restart** the server.
If you renamed dirs, edit the definition or symlinks.

Ownership check afterwards:

```bash
sudo -u gamesrv test -w /opt/gamesrv/worlds/minecraft-smp && echo "gamesrv can write ✓"
```

(Per the Bot Manager NFS note: test *behavior*, not displayed owner — NFS
UID mapping can lie.)

## Replacing the Minecraft server.jar

1. **Files** tab → area `install`, path `server.jar`, overwrite ✓ → upload
   the new jar.
2. **Control → Restart**.

Optional: back up the world first (**Backups → Back up world_dir now**) in
case the new jar migrates the world format irreversibly.

## Patching Palworld (or any SteamCMD game)

1. **Control → Update Game Software**. That re-runs
   `steamcmd +app_update 2394010 validate`, which patches in place.
2. **Restart** if the server was running before.
3. Your saves live in `world_dir` (symlinked to `Pal/Saved`), so they survive
   updates untouched.

## Editing a config file

1. **Files** → area `install`, path e.g.
   `Pal/Saved/Config/LinuxServer/PalWorldSettings.ini`.
2. Click **download**.
3. Edit locally.
4. Upload again with `path=Pal/Saved/Config/LinuxServer/PalWorldSettings.ini`
   and **overwrite** ticked.
5. **Restart** the server for changes to load.

## Moving a world in from another host

Two options:

- **Tarball route (recommended):** on the old host,
  `tar -czf world.tgz world/`. Upload to
  `worlds/_backups/minecraft-smp/imported-YYYYMMDD.tgz` on the LXC (SFTP is
  easiest), then in the UI Backups tab, click **restore** next to it. (Server
  must be stopped.)
- **File-by-file:** SFTP the world dir straight into `world_dir/`.

## Safety rails baked in

- Every file API call is scoped to `install_dir` or `world_dir` and rejects
  paths that escape via `..` or absolute paths (see `app/uploads.py`
  `_safe_join`).
- Uploads are streamed to `<name>.part` and atomically renamed.
- **Restore** requires the server to be stopped (409 otherwise) and refuses
  tarball entries that would escape the staging dir.
- Deleting the server root is refused explicitly.
- Backups live under `world_dir/../_backups/<name>/` — on TrueNAS, so ZFS
  snapshots protect them for free.
